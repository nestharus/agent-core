## 3. Data and behavior model summary

Beads is a local-first issue/task graph implemented as the `bd` CLI. The README describes the core model as a Dolt-backed structured memory with hash-based IDs, dependency tracking, ready-work detection, JSON output, graph links, hierarchy, messages, compaction, and persistent memories (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`). Its essential command set includes `bd ready`, `bd create`, `bd update <id> --claim`, `bd dep add`, `bd show`, `bd prime`, and `bd remember`, which overlaps the task selection, claim, dependency, inspection, and memory primitives wanted by a local agent-task substrate (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-problem-map.md`).

Beads stores state in `.beads/` by default. Embedded mode is the default and stores data under `.beads/embeddeddolt/` with file locking and single-writer semantics; server mode connects to an external `dolt sql-server`, stores data under `.beads/dolt/`, and supports multiple concurrent writers (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`). Beads also supports git-free usage through `BEADS_DIR` plus `bd init --stealth`; that bypasses git discovery and disables git hook installation and git operations (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`).

The behavior model is not literally "one file line equals one task." The visible task unit is a Beads issue record in Dolt/JSON-oriented storage, with JSON output and JSONL-like artifacts in the broader Gas Town architecture, not a canonical source-code line or markdown line primitive (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `https://raw.githubusercontent.com/gastownhall/gastown/main/docs/design/architecture.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-duplicates.md`). That makes Beads strong prior art for graph-backed local tasks, but not a direct answer to any future line-as-task contract until that contract exists (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-problem-map.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/risk/nes-186-risk-profile.md`).

Gas Town is broader than Beads. Its README describes a multi-agent workspace manager for coordinating Claude Code, GitHub Copilot, Codex, Gemini, and other agents, with Mayor coordination, rigs, worktrees, mailboxes, convoys, hooks, dashboards, telemetry, escalation, scheduler, and merge queue behavior (`https://raw.githubusercontent.com/gastownhall/gastown/main/README.md`). Its architecture document says Gas Town uses a two-level Beads architecture with town-level and rig-level `.beads/` stores, shared Dolt SQL Server storage, routes, redirects, and direct multi-agent writes to `main` via transaction discipline (`https://raw.githubusercontent.com/gastownhall/gastown/main/docs/design/architecture.md`). This dossier evaluates Beads as the local task graph candidate and treats Gas Town only as adjacent scope pressure.

## 4. Comparison axes

### Backend fit (alongside vs below Linear/Jira)

Evidence: Current `~/ai` ticket execution selects exactly one `ticket_system`, Jira or Linear, and active WU tickets live on the external tracker while `${scratch_dir}/ticket.md` is a rendered local copy (`/home/nes/ai/agents/implementation-pipeline-orchestrator.md`; `/home/nes/ai/workflows/implementation-pipeline.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-duplicates.md`). Beads provides local issue create/show/update/claim/close/dependency behavior, but it is a local Dolt-backed CLI/database surface, not an existing Jira/Linear peer in the orchestrator contract (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-hookpoints.md`).

Verdict: Beads fits best below or beside Linear/Jira as an inspiration source or local queue candidate, not as a peer backend until a future WU designs ticket-system pluggability, source-of-truth semantics, and publication rules (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/proposals/nes-186-NES-186.md`; `/home/nes/ai/DECISIONS.md`).

### Local/offline operation

Evidence: Beads supports embedded mode with no external server, stores data in `.beads/embeddeddolt/`, and supports git-free usage with `BEADS_DIR` plus `bd init --stealth` (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`). It also has server mode for multi-writer use, but that introduces an external Dolt SQL server and related host/port/socket/user/password environment variables (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`).

Verdict: Beads is strong local/offline prior art for a single-writer local task store, but its multi-writer story depends on server mode or Gas Town's server-managed architecture, which is a larger operational commitment (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `https://raw.githubusercontent.com/gastownhall/gastown/main/docs/design/architecture.md`).

### Files-as-tasks / line-as-task primitive

Evidence: The NES-186 planning evidence says no landed `agent-tasks`, `NES-180`, files-as-tasks, or line-as-task contract exists in `~/ai` today (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-problem-map.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-duplicates.md`). Beads exposes issue records, JSON output, dependency links, hierarchy, and Dolt-backed state rather than a literal source-line or markdown-line task primitive (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `https://registry.npmjs.org/@beads%2fbd`).

Verdict: Beads is useful model evidence for graph-backed local task records, but it should not be treated as satisfying a line-as-task primitive until the `agent-tasks` contract defines what a line means (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/risk/nes-186-risk-profile.md`).

### Connection-abstraction compatibility

Evidence: The connection abstraction defines `Client`, `Auth`, `System-conn`, and `Project-conn` for external service API or CLI provider account access, and says ordinary workflow scope, branch scope, worktree location, planning artifacts, issue IDs, and file paths are not automatically connections (`/home/nes/ai/conventions/connection-abstraction.md`). Beads is a local CLI plus project-local or `BEADS_DIR` storage, and it can run with no service Auth in embedded or stealth mode (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`).

Verdict: Model `bd` as a local CLI Client-like tool only if a future design needs that vocabulary; do not force `.beads/`, issue IDs, or a worktree path into `Project-conn` identity under the current convention (`/home/nes/ai/conventions/connection-abstraction.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-hookpoints.md`).

### Project/worktree and planning-layout safety (`.beads/`, `BEADS_DIR`, trunk/planning/worktree placement, git-diff implications)

Evidence: The project-layout convention places committed source in `trunk/`, planning artifacts outside the repo under `planning/<branch>/`, and branch worktrees under `worktrees/<branch>/`; it also warns that planning artifacts written under a worktree can pollute PR diffs (`/home/nes/ai/conventions/project-layout.md`). Beads defaults to `.beads/` in the project and says `bd init` creates or updates `AGENTS.md` by default, while `--skip-agents`, `--stealth`, custom agent files, and `BEADS_DIR` can reduce repo-level side effects (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`).

Verdict: Beads has usable escape hatches, but backend adoption needs an explicit storage-placement rule before implementation; the safest fit for `~/ai` would keep exploratory Beads state out of source diffs unless a later WU deliberately makes it a committed product surface (`/home/nes/ai/conventions/project-layout.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-hookpoints.md`).

### Data-model fit (issue fields, statuses, claim, dependency graph, hierarchy, comments, audit trail, memories, formulas/molecules, JSON output)

Evidence: Beads supplies ready work, create, claim, dependency add, show/audit-trail, prime, remember, hierarchy, graph links, JSON output, messages, compaction, and memories (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `https://registry.npmjs.org/@beads%2fbd`). Verdict: Beads' data model is richer than current visible `agent-tasks` needs and is strong inspiration for dependencies, claim, ready-work detection, and JSON CLI output; Gas Town's formulas, molecules, mail, and agent lifecycle state should not be bundled into a Beads-backend decision (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-duplicates.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/proposals/nes-186-NES-186.md`).

### Sync/conflict model (embedded/server Dolt, merge/conflict, multi-writer safety)

Evidence: Beads' embedded mode is single-writer with file locking, while server mode supports multiple concurrent writers through an external Dolt SQL server (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`). Gas Town centralizes Dolt data per town, uses one Dolt SQL Server, redirects worktree `.beads/` stores to canonical locations, and has agents write directly to `main` with transaction discipline (`https://raw.githubusercontent.com/gastownhall/gastown/main/docs/design/architecture.md`). `~/ai` currently relies on separate git worktrees for concurrent writers and treats merge as the join step (`/home/nes/ai/conventions/worktree-isolation.md`).

Verdict: Beads improves on ad hoc files only if the chosen mode matches the writer topology; embedded mode is too narrow for shared concurrent writers, and the server/Gas Town mode is too operationally broad to adopt without a dedicated backend-design WU (`https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `/home/nes/ai/conventions/worktree-isolation.md`).

## 8. Recommendation

use as inspiration

Rationale: Beads is a real, active, MIT-licensed, local-first graph issue tracker with strong primitives for ready work, claim, dependencies, hierarchy, JSON output, memories, and Dolt-backed storage (`https://github.com/gastownhall/beads`; `https://raw.githubusercontent.com/gastownhall/beads/main/README.md`; `https://registry.npmjs.org/@beads%2fbd`). It is not ready to become an authoritative `~/ai` backend now because current `~/ai` lacks a landed `agent-tasks` contract, has unresolved Jira/Linear abstraction and status-owner drift, keeps active WU tickets external, and has strict planning/worktree placement rules that Beads storage would need to respect (`/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-problem-map.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/research/nes-186-duplicates.md`; `/home/nes/ai/conventions/project-layout.md`; `/home/nes/ai/conventions/connection-abstraction.md`). This recommendation is Beads-only and does not adopt Gas Town orchestration (`https://raw.githubusercontent.com/gastownhall/gastown/main/README.md`; `/home/nes/projects/ai/planning/nes-186-gastown-beads-investigation/proposals/nes-186-NES-186.md`).
